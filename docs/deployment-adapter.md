# The deployment adapter

The deployment adapter is the seam that lets the platform target **Coolify
today and a different orchestrator tomorrow** (AWS ECS, Kubernetes) without
changing any provisioning, billing, or catalog code (PRD §8.3, §2.1 goal 7,
NFR "Portability"). It is the single most important abstraction in this repo.

**The rule:** the `provisioning` service and its Taskiq tasks call only the
`DeploymentAdapter` Protocol. They never import a Coolify SDK, never construct
an HTTP client, never reference orchestrator-native concepts (Coolify
"applications", K8s "manifests"). All of that lives behind an adapter.

**The acceptance test for the abstraction:** the exact same convergence code
must run, unchanged, against an in-memory `FakeDeploymentAdapter` and the real
`CoolifyAdapter`. If a behavior change is needed to swap adapters, the
abstraction has leaked and must be fixed.

## The port

`ports/deployment_adapter.py` — a `Protocol`, async, one v1 adapter per
orchestrator plus a fake for tests. The InstanceHandle returned by
`create_instance` is **opaque** to the domain (persisted as
`instance.deployment_handle` JSONB) and only ever passed back to the same
adapter.

```python
@runtime_checkable
class DeploymentAdapter(Protocol):
    """Orchestrator-agnostic lifecycle of one customer's Odoo deployment.

    Every method is idempotent: re-invoking with the same spec/handle
    converges to the desired state rather than duplicating resources. The
    domain relies on this to make retries safe.
    """

    async def create_instance(self, spec: InstanceSpec) -> InstanceHandle: ...
    async def update_instance(self, handle: InstanceHandle, spec: InstanceSpec) -> InstanceHandle: ...
    async def suspend_instance(self, handle: InstanceHandle) -> None: ...
    async def reinstate_instance(self, handle: InstanceHandle) -> None: ...
    async def delete_instance(self, handle: InstanceHandle) -> BackupRef | None: ...
    async def get_instance_status(self, handle: InstanceHandle) -> DeploymentStatus: ...
    # optional, capability-gated:
    async def stream_instance_logs(self, handle: InstanceHandle) -> AsyncIterator[bytes]: ...
```

Notes:

- `create_instance` provisions container set + database + volumes + network +
  installs the base + entitled modules and returns the handle. Returning
  successfully means "resources exist"; readiness is confirmed separately via
  `get_instance_status` (the state machine waits for healthy before
  `configuring`).
- `delete_instance` performs the final backup (or returns a `BackupRef` the
  caller records) before tearing down; PRD §10.4. Backups are a capability of
  the adapter (PRD open-question 4) with a default per orchestrator.
- `suspend_instance` applies **soft** suspension in MVP (login-blocked, still
  running). A future `policy` parameter selects hard suspension.
- `stream_instance_logs` is optional; adapters that can't stream raise
  `NotImplementedError` and the operator log view degrades gracefully.

## InstanceSpec — the orchestrator-agnostic desired state

`modules/provisioning/spec.py` builds an `InstanceSpec` from the entitlement
picture (module set, seat cap, resource caps — see [events.md](events.md)
§Resolving entitlements). The spec is a frozen dataclass; each adapter
translates it into its native concepts.

```python
@dataclass(frozen=True, slots=True)
class InstanceSpec:
    subscription_id: UUID
    customer_id: UUID
    slug: str                       # hostname label → {slug}.{INSTANCE_DOMAIN_SUFFIX}
    admin_email: str
    odoo_image: str                 # base image ref incl. the entitlements plugin
    module_set: tuple[str, ...]     # entitled Odoo module identifiers
    seat_cap: int
    resource_caps: Mapping[str, int]   # e.g. {"storage_gb": 50, "email_per_day": 5000}
    env: Mapping[str, str]          # non-secret env for the Odoo container
    resources: ResourceRequests     # cpu / memory requests + limits
    # secrets (db password, admin password) are generated by the adapter and
    # returned via the create result, never placed in the spec.
```

Keeping `InstanceSpec` free of orchestrator concepts (no Coolify ids, no K8s
kinds) is what makes the abstraction hold.

## v1 adapters

### `FakeDeploymentAdapter` (milestone 1 — the default for all of milestone 1)

An in-memory implementation that records calls and transitions a simulated
instance through deploying → healthy on a short timer (or instantly, with an
injected `Clock`). It lets the **entire** provisioning pipeline — consumer,
state machine, tasks, outbox, events — be built and unit-tested with **no
Coolify and no real Odoo**. It is also the basis for:

- `tests/` unit coverage of every convergence path (create, update, suspend,
  reinstate, deprovision, failure + retry).
- a fault-injection mode (`fail_on={"create"}`, latency, partial-failure) to
  exercise the retry/backoff and `instance.failed` paths deterministically.

This mirrors the "stub deployment adapter so flows requiring a ready instance
can advance without Coolify" that customer-portal already assumes in its E2E
Compose stack.

### `CoolifyAdapter` (milestone 2 — design sketch, not yet built)

> **Status: undefined / greenfield.** There is **no** Coolify configuration
> anywhere in the platform yet (`platform-infra` lists "Coolify project
> definitions" under *What's NOT here yet*; there is no API base URL, no token,
> no Odoo stack template). This section is a design sketch to be validated by a
> dedicated **Coolify-API spike** at the start of milestone 2 — the platform's
> #1 risk (PRD §16) is exactly Coolify API gaps around volumes/networking/
> suspension, so the spike prototypes the riskiest operations first.

Anticipated shape:

- An `aiohttp` client against the Coolify API at `COOLIFY_API_URL` with a
  bearer `COOLIFY_API_TOKEN` (both injected via settings; sourced from the
  orchestrator's secret store).
- Maps one `InstanceSpec` onto Coolify resources: a project/application for the
  Odoo container, a managed Postgres for that instance's database, persistent
  volumes, a domain/route for `{slug}.{INSTANCE_DOMAIN_SUFFIX}`, and env/secret
  injection.
- `InstanceHandle` = the Coolify project/application/database ids (opaque JSONB
  on `instance.deployment_handle`).
- Open questions the spike must answer (tracked in milestone-2 planning):
  per-instance Postgres provisioning (Coolify-managed vs a DB created in the
  shared cluster), volume lifecycle on suspend/delete, how the Odoo base image
  (with the `platform_entitlements` plugin) is built and where it's hosted
  (likely GHCR), and how Coolify health maps onto `DeploymentStatus`.

### Future adapters

`KubernetesAdapter` (Helm chart for the Odoo stack) and `AWSAdapter`
(ECS+RDS / EKS) implement the same Protocol. Adding one must not touch
`modules/provisioning/`.

## Suspension and backup policies

- **Suspension** — MVP is soft only (login blocked, data retained, container
  running; cheaper to reinstate). Hard suspension (scale-to-zero, volume
  retained) is a later capability behind a policy flag (PRD §10.3).
- **Backup / deprovision** — on cancel + grace + export window: generate the
  final export, `delete_instance` returns/records a `BackupRef`, retain the
  snapshot for the compliance window (e.g. 30 days) then purge (PRD §10.4).
  Backup is an adapter capability with a per-orchestrator default
  implementation (PRD open-question 4).

## Why this matters here and not in platform-api

platform-api deliberately owns **no** provisioning code — it publishes
`subscription.*` events and reads the `provisioning.instance` registry. All
deployment concerns, and therefore this entire abstraction, live in this repo
(platform-api `docs/architecture.md`: "No Odoo provisioning. That belongs to
`provisioning-worker`"). That is what keeps billing/catalog/subscription logic
independent of the orchestrator choice.
