<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ENV → Settings | Untrusted env vars (string inputs) cross into typed Settings via pydantic-settings |
| .env file → git | .env file must never be committed; .gitignore is the barrier |
| Docker build context → image | .dockerignore prevents .env and .venv from entering the build context |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | .env file | mitigate | .env is listed in .gitignore and .dockerignore; .env.example commits safe defaults only |
| T-01-02 | Information Disclosure | Dockerfile build ARGs | accept | No secrets are passed as Docker ARGs; secrets arrive via env_file: .env in Compose (D-12/D-13) |
| T-01-03 | Tampering | Dependency supply chain | accept | All packages are well-established (Package Legitimacy Audit in RESEARCH.md); uv.lock pins exact versions preventing silent upgrades |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | All packages in RESEARCH.md Package Legitimacy Audit are marked Approved; no [ASSUMED]/[SUS] packages require human gate |
</threat_model>
