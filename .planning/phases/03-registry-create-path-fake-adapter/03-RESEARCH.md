- D-02: Introduce an `EntitlementResolver` port now. A `Protocol` in `ports/` with a default M1 implementation; M2 swaps in the cross-schema read-back adapter without touching `spec.py`. One-line wiring change in `main.py`.
- D-03: Settings defaults populate the placeholder. `odoo_image`, `instance_domain_suffix`, `default_seat_cap`, `default_resource_caps` (JSON), `module_set` (empty or small base set). Do NOT invent a `line_count → seat_cap` mapping.

**Convergence execution (PROV-02)**
- D-04: One `create` Taskiq task drives all edges. A single `create` `provisioning_task` walks `pending → deploying → configuring → ready`; on failure the same task re-runs idempotently.
- D-05: Handler opens, the job converges. `handle_subscription_activated` (inside Phase-2 dedupe `session_scope()` txn): insert `instance` row (`pending`) + `provisioning_task` (`pending`) + `processed_event` row, commit, then enqueue the Taskiq `create` job. The Taskiq job performs ALL adapter calls and transitions.
- D-06: Injected `Clock` + poll `get_instance_status`. The job polls `adapter.get_instance_status(handle)` until healthy with clock-driven waits; the fake transitions `deploying → healthy` instantly so unit tests are deterministic with no real sleep. Add a `Clock` port if not already present.
- D-07: Write a minimal v1 `enforcement_snapshot` at `configuring`. One row (`module_set`/`seat_cap`/`resource_caps` from spec, `version=1`), set `instance.snapshot_version`. Recompute/version-bump logic is Phase 5.

**Retry & failure semantics (PROV-04)**
- D-08: Settings-tunable exponential backoff. `max_attempts`, `base_delay_s`, `multiplier`, `cap_s` as typed `Settings`. Proposed defaults: 5 attempts / 2s base / ×2 / 60s cap (→ 2, 4, 8, 16, 32s).
- D-09: `failed` on each attempt, re-enter on retry. Each failed step sets `instance.status=failed` + `failed_step` + `failure_reason`; the scheduled retry transitions instance back into `deploying` and clears failure fields on success. Note: `instance.failed` event emission is Phase 4+; Phase 3 only sets DB columns/status.
- D-10: `provisioning_task` is the durable backoff ledger. `attempt_count` / `max_attempts` / `next_attempt_at` / `last_error` persisted in the converging transaction; retry is a delayed Taskiq re-kick to `next_attempt_at`. Do NOT rely on Taskiq's in-memory retry middleware.

**Credential delivery (PROV-08)**
- D-11: Adapter generates secrets, returns them in the create-result. Fake adapter returns a stable secret for the same spec (idempotent re-run consistency). Same contract for the real Coolify adapter in M2.
- D-12: In-memory pass-through, never persisted. The `create` job carries plaintext secret in-memory from adapter result straight to `NotificationTransport`, then drops it. Never written to any column, never in any event, never through structlog.
- D-13: Deliver once, guarded on first `ready_at`. Send credentials exactly when instance first transitions to `ready` (`ready_at` goes `null → now`). A re-converge or retry reaching `ready` again must NOT re-notify.

## Summary

Phase 3 extends the existing Phase 2 skeleton — a working consumer with no-op handlers — into a complete create-path convergence loop. The Phase-2 CONS-03 IntegrityError gap is confirmed **closed** (verified in `02-VERIFICATION.md`, re-verified 2026-06-02). The Phase-3 work requires: (1) three new Alembic-managed tables in the `provisioning` schema; (2) four new ports (`DeploymentAdapter`, `NotificationTransport`, `EntitlementResolver`, `Clock`) and three new adapters (`FakeDeploymentAdapter`, `ConsoleNotificationTransport`, M1 default `EntitlementResolver`); (3) completing `spec.py`, `repository.py`, `service.py`, `tasks.py` in `modules/provisioning/`; (4) wiring the real body of `handle_subscription_activated`; and (5) updating `main.py` and `settings.py`.

The critical design insight is the **transaction boundary split**: the handler inserts instance + task rows inside the Phase-2 dedupe `session_scope()` transaction (same commit as `processed_event`), then **after** commit enqueues the Taskiq `create` job. All adapter calls (which are slow and can fail) run inside the Taskiq task, not the consumer loop. Durability for retry is achieved by persisting `next_attempt_at` to the DB and using an `asyncio.sleep`-then-kick pattern within the convergence concern — no external scheduler process needed.

For status columns, the recommended approach is a hand-reviewed Alembic migration using `op.execute()` to create the Postgres ENUM types explicitly in the `provisioning` schema before the tables. Autogenerate produces the right column references but drops the `CREATE TYPE` DDL and loses CHECK constraints; the migration must be manually corrected before committing.

**Primary recommendation:** Deliver Phase 3 in four parallel-capable waves: (1) DB tables + Alembic migration, (2) ports + adapters, (3) spec builder + convergence service + tasks, (4) handler wiring + settings + main.py + tests.

### Pattern 1: Same-Transaction Handler → Enqueue

The handler runs inside the Phase-2 dedupe `session_scope()` transaction already in flight. It adds ORM objects to that session and commits. Taskiq enqueue happens **after** commit, not inside:

```python
# handlers.py — handle_subscription_activated
async def handle_subscription_activated(raw_env, payload, session: AsyncSession) -> None:
    # session is already open (dedupe wrapper owns it)
    structlog.contextvars.bind_contextvars(
        envelope_id=raw_env.id,
        subscription_id=str(payload.subscription_id),
        correlation_id=raw_env.correlation_id,
        # instance_id bound after INSERT
    )
    spec = entitlement_resolver.resolve(payload, settings)
    instance = Instance(
        id=uuid7(),
        subscription_id=payload.subscription_id,
        customer_id=payload.customer_id,
        status=InstanceStatus.pending,
        admin_email=...,  # from spec or payload
        version=1,
    )
    task = ProvisioningTask(
        id=uuid7(),
        instance_id=instance.id,
        task_type=TaskType.create,
        status=ProvisioningTaskStatus.pending,
        source_event_id=raw_env.id,
        attempt_count=0,
        max_attempts=settings.provisioning_max_attempts,
        payload=spec.to_dict(),
    )
    session.add(instance)
    session.add(task)
    structlog.contextvars.bind_contextvars(instance_id=str(instance.id))
    # processed_event insert happens in dedupe wrapper AFTER this returns
    # commit happens in dedupe wrapper AFTER processed_event is staged
    # → at that point, control returns to the wrapper which commits ALL 3 rows atomically
    # → THEN we enqueue (after commit, outside this function)
```

**Critical:** Taskiq enqueue must happen in the dedupe wrapper **after** the final `session.commit()`, not inside the handler body. The handler only stages ORM objects. The actual `broker.task.kiq(instance_id, task_id)` call happens after commit.

One implementation approach: the handler returns the pending `(instance, task)` and the wrapper calls enqueue after commit. Or: the handler stores a "pending enqueue" closure on a context var that the wrapper drains after commit.

The cleaner pattern (avoids context var gymnastics): make `handle_with_dedupe` aware of a post-commit hook, or have the handler itself enqueue inside a `try` block post-commit. Since the handler is called from inside the session scope, the real solution is to enqueue from within the handler body right before returning — at the point the handler is called, `session.commit()` has NOT happened yet (the wrapper calls commit after the handler returns). So the handler should NOT enqueue inline.

**Resolved pattern:** The handler returns a sentinel object or raises a special value signalling "enqueue this after commit", or more cleanly: the handler body only does DB ops; `make_handler_registry` or `handle_with_dedupe` is extended to accept a post-commit callback that the handler registers. The simplest approach: wrap the handler with a post-commit queue — after `session.commit()`, drain any registered post-commit callbacks. This keeps the dedupe wrapper's invariant.

**Simpler still:** The handler stores (instance_id, task_id) in a thread-local / contextvars slot. The dedupe wrapper, after commit, reads that slot and kicks. This is consistent with how structlog's `bind_contextvars` already works in this codebase.

### Pattern 2: Taskiq Create Task — Convergence Steps

```python
# tasks.py
@broker.task
async def create_instance_task(instance_id: str, task_id: str) -> None:
    """Drive pending → deploying → configuring → ready for one instance."""
    # 1. Load instance + task from DB
    # 2. Call adapter.create_instance(spec) → handle + secrets (CreateResult)

    # 4. Poll get_instance_status(handle) with clock.sleep between polls
    # 5. Update instance: status=configuring
    # 6. Write enforcement_snapshot (version=1)
    # 7. Update instance: status=ready, ready_at=now
    # 8. Update task: status=succeeded
    # 9. If ready_at was previously null: send credentials via transport
    # [secrets never logged or written to DB]
```

### Pattern 3: Durable Backoff Retry

The retry mechanism avoids adding a 5th scheduler concern. The pattern is:

```python
# Inside create_instance_task on failure:
try:
    await _do_convergence_steps(...)
except DeploymentFailed as exc:
    delay = _compute_backoff(attempt_count, settings)  # 2s * 2^attempt, capped at 60s
    next_attempt_at = clock.now() + timedelta(seconds=delay)
    await repository.record_failure(task_id, instance_id, exc, next_attempt_at)
    if attempt_count < settings.provisioning_max_attempts:
        await asyncio.sleep(delay)  # blocking within the Taskiq worker slot
        await create_instance_task.kiq(instance_id, task_id)
    # if max attempts reached: leave task.status=failed (terminal)
    return
```

**Note on `asyncio.sleep` inside Taskiq tasks:** The `RedisStreamBroker` worker loop (`listen()`) fetches the next task message only when the current one finishes. A sleeping task occupies one worker slot. For this phase (single-instance worker, small scale), this is acceptable. The `provisioning_task.next_attempt_at` column serves as the durable record — if the worker restarts mid-sleep, the boot sequence can query for overdue tasks and re-kick them. The CONTEXT.md D-10 says "delayed Taskiq re-kick" — this is the implementation.

**Alternative for worker restart recovery:** `_run_convergence` in `main.py` should, on boot, query for `provisioning_task` rows in `pending` or `failed` status with `next_attempt_at <= now()` and kick them. This closes the durability gap on restart.

### Pattern 4: FakeDeploymentAdapter Fault Injection

```python
# adapters/fake_deployment.py
@dataclass
class FakeDeploymentAdapter:
    fail_on: set[str] = field(default_factory=set)  # e.g. {"create"}
    fail_count: int = 1  # fail this many times before succeeding
    _call_counts: dict[str, int] = field(default_factory=dict, init=False)
    _instances: dict[str, DeploymentStatus] = field(default_factory=dict, init=False)

    async def create_instance(self, spec: InstanceSpec) -> CreateResult:
        count = self._call_counts.get("create", 0)
        self._call_counts["create"] = count + 1
        if "create" in self.fail_on and count < self.fail_count:
            raise DeploymentFailed("injected failure")
        handle = InstanceHandle(id=str(spec.subscription_id))
        self._instances[handle.id] = DeploymentStatus.HEALTHY
        return CreateResult(
            handle=handle,
            admin_password="test-password-stable",  # stable per spec for idempotency
            db_password="test-db-password-stable",
        )

    async def get_instance_status(self, handle: InstanceHandle) -> DeploymentStatus:
        return self._instances.get(handle.id, DeploymentStatus.DEPLOYING)
```

### Anti-Patterns to Avoid

- **Enqueuing inside the session scope** — if the commit fails after enqueue, the job runs against a row that doesn't exist yet. Always enqueue after commit.
- **Logging credentials** — `log.info("credentials ready", password=result.admin_password)` is the secret-leak trap. Never bind password/token fields to structlog.
- **Calling `adapter.create_instance` from `service.py` directly** — adapter calls go in `tasks.py` (retryable Taskiq jobs), not the service/handler. The service drives state transitions; tasks drive adapter calls.
- **Using Postgres pg_random_uuid()** — this gives uuid v4, not v7. Generate `uuid.uuid7()` in Python as the `default=` on the SQLAlchemy column.
- **Using `from __future__ import annotations`** — Python 3.14 has PEP 649; the import is forbidden in this project, including in generated Alembic files.
- **Using `asyncpg`** — the stack pins `psycopg[binary]` 3.3. Do not introduce asyncpg.
- **Relying on Taskiq's built-in retry middleware** — D-10 explicitly forbids this. The `provisioning_task` ledger IS the retry state; Taskiq is just the job queue.
- **Using Postgres ENUM without schema qualification** — Alembic autogenerate will create `instance_status` in the default `public` schema unless `schema='provisioning'` is passed to the ENUM type.

### Alembic Migration Approach

The Phase 2 migration (`0e3f3be0f9ad`) created `processed_event`. Phase 3 adds a single new revision that:

1. Creates three Postgres ENUM types via `op.execute()` (autogenerate does NOT do this correctly for schema-qualified types):
   ```sql
   CREATE TYPE provisioning.instance_status AS ENUM ('pending', 'deploying', 'configuring', 'ready', 'suspended', 'failed', 'deprovisioning', 'deprovisioned');
   CREATE TYPE provisioning.task_type AS ENUM ('create', 'update', 'suspend', 'reinstate', 'delete');
   CREATE TYPE provisioning.task_status AS ENUM ('pending', 'running', 'succeeded', 'failed');
   ```
2. Creates `instance` table with UUID PK (gen by Python), ENUM columns referencing the above types, JSONB columns, UNIQUE on `subscription_id`.
3. Creates `provisioning_task` table with FK to `instance`, UNIQUE (instance_id, change_set_id).
4. Creates `enforcement_snapshot` table with FK PK to `instance`.
5. `downgrade()` drops tables then drops ENUM types (reverse order).

**Workflow:** `make revision name="add instance tables"` — review the autogenerated file, replace the ENUM column stubs with `op.execute()` DDL for the type creation, verify the `CREATE TABLE` DDL includes the UNIQUE constraints and FKs. Check for any missing `schema="provisioning"` arguments.

**No `from __future__ import annotations`** in the migration file (project-wide rule; the template must not have it; wipe it if autogenerate inserts it).

## Convergence Service & Taskiq

### Broker Registration in `main.py`

The existing `_run_convergence` concern boots the `RedisStreamBroker` and waits for shutdown. Phase 3 extends it to also call `broker.listen()` to consume task messages. The tasks must be imported (so the `@broker.task` decorator runs and registers them) before `broker.startup()`.

```python
# main.py _run_convergence — Phase 3 extension
from provisioning_worker.modules.provisioning import tasks as provisioning_tasks  # registers tasks

async def _run_convergence(settings: Settings, shutdown: asyncio.Event) -> None:
    broker = RedisStreamBroker(url=str(settings.valkey_url))
    await broker.startup()
    log.info("taskiq broker connected")
    # Start the task listener (replaces the bare shutdown.wait())
    async with asyncio.TaskGroup() as tg:
        tg.create_task(_listen_tasks(broker, shutdown))
    await broker.shutdown()

async def _listen_tasks(broker: RedisStreamBroker, shutdown: asyncio.Event) -> None:
    async for msg in broker.listen():
        if shutdown.is_set():
            break
        await broker.receiver.callback(msg.data)
        await msg.ack()
```

**Worker boot recovery:** Before starting the listener, query for `provisioning_task` rows with `status IN ('pending', 'failed') AND next_attempt_at <= now()` and re-kick them. This handles the restart-mid-sleep durability gap from D-10.

### Backoff Formula

```python
def _compute_backoff_seconds(attempt_count: int, settings: Settings) -> float:
    """Exponential backoff: base * multiplier^attempt, capped at cap_s."""
    delay = settings.provisioning_base_delay_s * (settings.provisioning_multiplier ** attempt_count)
    return min(delay, settings.provisioning_cap_s)
```

Default sequence for 5 attempts, 2s base, ×2, 60s cap: 2, 4, 8, 16, 32 (all under cap).

## Common Pitfalls

### Pitfall 1: Enqueue Inside Session Scope → Phantom Jobs
**What goes wrong:** If `broker.task.kiq(...)` is called before `session.commit()`, and the commit subsequently fails (e.g., unique constraint on `subscription_id` due to duplicate delivery), the job fires against a non-existent instance row and crashes.
**Why it happens:** The Phase-2 dedupe wrapper calls the handler first, then stages the `processed_event` insert, then commits. Any code the handler calls before returning is still pre-commit.
**How to avoid:** The handler must not call `broker.task.kiq()` directly. Use a post-commit hook pattern: the handler registers `(instance_id, task_id)` in a contextvars slot; the dedupe wrapper drains it after `session.commit()` succeeds.
**Warning signs:** Tests that mock `session.commit` to raise and still observe the broker being kicked.

### Pitfall 2: Alembic Drops Schema-Qualified ENUM Types
**What goes wrong:** Autogenerate with `include_schemas=True` sometimes generates `sa.Enum(...)` columns without referencing a named type, or generates a `DROP TYPE` in downgrade that fails because the type is in a different schema than Alembic expects.
**Why it happens:** SQLAlchemy autogenerate does not reliably handle Postgres ENUM types in non-public schemas. The `CREATE TYPE` DDL is emitted (or not) incorrectly.
**How to avoid:** After `make revision name="..."`, inspect the generated file. Replace any autogenerated ENUM column definitions with hand-written `op.execute("CREATE TYPE provisioning.instance_status AS ENUM (...)")` DDL in `upgrade()`. Use `existing_type=` in `op.alter_column` if modifying later.
**Warning signs:** `migrate` fails with `type "instance_status" does not exist` or `CREATE TYPE provisioning.instance_status already exists`.

### Pitfall 3: Secret Leak Through structlog
**What goes wrong:** `log.info("instance ready", admin_password=result.admin_password)` writes credentials to the JSON log stream. Even `log.debug(...)` is dangerous.
**Why it happens:** structlog kwargs are serialized verbatim. Developers under time pressure add debugging log lines with full result objects.
**How to avoid:** D-12: secrets are dropped immediately after `notification_transport.send_credentials(...)`. Never bind `admin_password`, `db_password`, or any field named `*_password`, `*_token`, `*_secret` to structlog. Review every `log.*` call in `tasks.py` for credential fields before committing.
**Warning signs:** `grep -r 'password' src/ | grep log` produces results.

### Pitfall 4: `ready_at` Guard Not Checked → Double Credential Delivery
**What goes wrong:** If a retry reaches `ready` after a previous attempt already set `ready_at`, the notification fires twice (two emails with the same credentials).
**Why it happens:** The convergence task unconditionally calls `transport.send_credentials()` when transitioning to `ready`.
**How to avoid:** D-13: Before calling `transport.send_credentials()`, read `instance.ready_at`. Only send if `ready_at IS NULL` (i.e., this is the first successful transition). Read `ready_at` inside the same DB transaction that sets it to `now()` — the transition and the guard are atomic.
**Warning signs:** Integration test counting `ConsoleNotificationTransport` calls where `fail_count=1` sees 2 calls instead of 1.

### Pitfall 5: `asyncio.sleep` Blocking the Consumer Loop
**What goes wrong:** If the backoff sleep happens in the consumer loop instead of the Taskiq worker, `XREADGROUP` stops polling for the sleep duration and the worker appears to stall.
**Why it happens:** Confusing the Taskiq worker concern with the Valkey Streams consumer concern.
**How to avoid:** `asyncio.sleep` for backoff ONLY inside the Taskiq task function (which executes in `_run_convergence`, not `_run_consumer`). The Valkey consumer (`_run_consumer`) never sleeps longer than `_POLL_COUNT_BLOCK_MS` (1 second).
**Warning signs:** Integration test sees the second delivery of a duplicate event taking much longer than expected.

### Pitfall 6: Forgetting `create_type=False` on Mapped Enum Columns
**What goes wrong:** SQLAlchemy tries to `CREATE TYPE provisioning.instance_status` when the table is created (via `Base.metadata.create_all` in `pg_engine` test fixture), but the migration also creates it — one of them will fail with "type already exists".
**Why it happens:** Default behavior of `PG_ENUM` is `create_type=True`.
**How to avoid:** Use `PG_ENUM(..., create_type=False)` on all ORM-mapped ENUM columns. Let the migration own the DDL. In test fixtures, the `pg_engine` fixture uses `Base.metadata.create_all` which calls SQLAlchemy's table creation path — without `create_type=False`, it will attempt to create the ENUM type again.
**Warning signs:** `create_all` fails with `ERROR: type "instance_status" already exists`.

### Pitfall 7: `uuid.uuid7` vs `uuid.uuid4` as PK Default
**What goes wrong:** `mapped_column(UUID, primary_key=True, default=uuid.uuid4)` gives v4 UUIDs; docs spec v7.
**Why it happens:** `uuid.uuid4` is the well-known function; `uuid.uuid7` is new in Python 3.14.
**How to avoid:** Explicitly import `from uuid import uuid7` and use `default=uuid7` on UUID PKs. Verify with `assert instance.id.version == 7` in a test.
**Warning signs:** `test_models.py` (if it checks UUID version) fails.

### Pitfall 8: Taskiq Dependency Context Not Wired Before Task Registration
**What goes wrong:** Tasks annotated with `TaskiqDepends()` fail with a "dependency not found" error at runtime because `broker.add_dependency_context(...)` was not called before `broker.startup()`.
**Why it happens:** The `@broker.task` decorator runs at import time; the dependency context must be populated before any task is kicked.
**How to avoid:** In `main.py`, wire `broker.add_dependency_context(...)` immediately after constructing the broker and before calling `broker.startup()`. Import `tasks.py` after that so the decorator runs with a fully wired broker.

### Known Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Secret in log line | Information disclosure | Never pass `*_password`/`*_token` as structlog kwargs; Console transport writes to `sys.stdout` directly, not through logger. |
| Duplicate provisioning (re-delivered `activated`) | Denial of service (resource duplication) | `subscription_id UNIQUE` on `instance` table raises `IntegrityError`; handler dedupe via `processed_event` + `handle_with_dedupe` wraps it. |
| Credential written to DB column | Tampering/disclosure | D-12 + D-11: secrets never stored; `instance_credential` table not created this phase. |
| Uncaught exception crashes consumer | DoS | `_dispatch` boundary catches all handler exceptions; task failures update DB state and do not propagate to the consumer loop. |
| SQL injection via JSONB payload | Tampering | SQLAlchemy ORM parameterizes all queries; no raw string interpolation in SQL. |

## Open Questions (RESOLVED)

All three questions are resolved below and their decisions are adopted by the Phase-3 plans.

1. **Post-commit enqueue mechanism in `handle_with_dedupe`** — **RESOLVED.**
   - Decision: **Use a `ContextVar` slot** (consistent with how structlog contextvars already work in this codebase). The handler stores the enqueue request `(task_name, args)` in a `ContextVar`; the `handle_with_dedupe` wrapper drains it **after** `session.commit()` returns and only then calls `.kiq(...)`. This keeps handler and wrapper decoupled and guarantees a job is never enqueued against an uncommitted (or rolled-back) row. Adopted in Plan 03-03 (extends `shared/event_consumer.py` + handler body).

2. **Taskiq task listener integration in `_run_convergence`** — **RESOLVED (API verified against installed `taskiq==0.12.4`).**
   - Verified facts (via `inspect` on the installed package):
     - `AsyncBroker.listen(self) -> AsyncGenerator[bytes | AckableMessage, None]` exists and is the worker entry point; it yields incoming messages and auto-acks `AckableMessage` after processing.
     - There is **no** `broker.receiver` attribute — the earlier assumption A2 (`broker.receiver.callback(msg.data)`) was **incorrect** and must not be used.
     - The correct low-level API is `taskiq.receiver.Receiver(broker, run_startup=False, ...)` whose `async callback(self, message: bytes | AckableMessage, raise_err: bool = False) -> None` parses the message, dispatches to the registered task, and saves the result.
   - Decision: In the convergence concern, construct one `Receiver(broker, run_startup=False)` and run `async for message in broker.listen(): await receiver.callback(message)`. Tasks must be imported/registered before `broker.startup()`. This composes cleanly with the existing `asyncio.TaskGroup` in `main.py`. (`Receiver.listen(finish_event)` is the all-in-one alternative if a manual loop is undesirable.) Adopted in Plan 03-03 (`main.py` wiring / convergence concern).

3. **`ResourceRequests` dataclass shape** — **RESOLVED.**
   - Decision: Define a minimal `ResourceRequests(frozen=True, slots=True)` value object with string fields defaulted from `Settings` (e.g. `cpu_request: str = "0.5"`, `memory_request: str = "512Mi"`). The `FakeDeploymentAdapter` ignores it; the real Coolify adapter consumes it in M2. It lives alongside `InstanceSpec` (see the dependency-direction note below). Adopted in Plan 03-01.

> **Dependency-direction correction (consumed by the plans):** the value objects that cross the `DeploymentAdapter` port boundary — `InstanceSpec`, `ResourceRequests`, `InstanceHandle`, `CreateResult` — must be **defined at the port (or in `shared/`), not in `modules/provisioning/spec.py`**, because `ports/` must never import from `modules/` (CLAUDE.md §4). `spec.py` holds only the **builder** and imports these types from the port/shared location. The earlier code snippets that imported `InstanceSpec` from `modules/...spec.py` into `ports/deployment_adapter.py` are superseded by this correction.
