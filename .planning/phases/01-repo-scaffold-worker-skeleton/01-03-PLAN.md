<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Docker build context → image layers | .dockerignore must exclude .env, .venv, secrets from the build context |
| docker-compose env_file → container | Secrets arrive via env_file: .env (never as Docker ARG or hardcoded ENV) |
| CI runner → Docker registry | No push in M1; image stays local/CI ephemeral — no registry credential exposure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11 | Information Disclosure | .env baked into Docker image | mitigate | .dockerignore excludes .env and .env.* (written in Plan 01); env_file: .env in Compose passes secrets at runtime, not build time |
| T-01-12 | Elevation of Privilege | Container running as root | mitigate | Runtime stage creates user platform uid/gid 10001 and sets USER platform before ENTRYPOINT (D-12) |
| T-01-13 | Information Disclosure | Docker ARG leaking secrets in image layers | accept | No secrets are passed as Docker ARGs; only uv/Python tool versions appear in ARG (if any); env_file: is the secrets boundary |
| T-01-14 | Tampering | CI pushing to registry without review | mitigate | push: false in build-push-action (D-15); no registry credentials configured in M1; build proves image builds, nothing more |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | CI uses uv sync --frozen from committed uv.lock; no new package installs in this plan |
</threat_model>
