<objective>
Close CONS-03 gap: harden `handle_with_dedupe` to catch the `IntegrityError` the composite PK is designed to raise under concurrent/reclaim-race duplicate delivery, and add a handler-failure error boundary in `_dispatch` so handler exceptions never crash the poll loop.

Purpose: Under at-least-once delivery the current SELECT-then-INSERT guard raises unhandled on a committed duplicate, crashing the consumer task with no XACK — a denial-of-availability today and an exactly-once hole when Phase 3 adds real side-effects. The fix is two targeted code changes and one regression test.

Output: `handle_with_dedupe` treats an `IntegrityError` on `commit()` as an idempotent short-circuit (rollback + return). `_dispatch` catches unexpected handler exceptions, logs them at error, and returns without XACK so XAUTOCLAIM reclaims the message later. A new integration test (`test_concurrent_duplicate`) drives the conflict deterministically and asserts no exception is raised.

Scope: This plan addresses CR-01 (CONS-03 blocker) and WR-01 (handler-failure boundary, same propagation path) as must-do. WR-03 (parenthesize `except (json.JSONDecodeError, KeyError):`) and WR-04 (log missing-handler warning) are folded in as low-risk hardening because they touch the same `_dispatch` block. WR-02 (env.py version-table drift), WR-05 (shutdown not observed in reclaim scan), and all Info findings are deferred — they do not touch the files in scope and do not block CONS-03.
</objective>
